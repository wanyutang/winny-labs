## 討論議題

- 註解語言該用英文還是中文
	- 團隊需有共識：英文程度須達到所有人都能理解。
	- Java 專有名詞可保留英文原文，不必硬翻中文。
	- 大型專案與外包合作時，可加入必要母語註解補充，提高維護效率。
- 變數命名 vs 註解
	- 命名再好也無法完全取代註解，因為不同工程師的命名標準不一致。
	- 乙方環境需對應客戶功能稱呼，命名與註解要能協助 QA、新人、外包快速對應功能與需求。
	- 良好命名是基本，必要註解仍不可省略。
- 規格文件 vs 註解程式邏輯 
	- 規格文件版本與程式版本常不同步，程式內的註解才是「當下版本」的真實邏輯。
	- 註解要能說明程式邏輯的目的，讓開發者在看程式時能立即理解。
	- 程式註解需與程式同步維護，不可依賴規格文件取代。
- API req res 與 entity 的欄位註解 
	- 可參考 swagger 用 schema 標註的方式，讓註解與程式結構合一。
	- API 的 function 與欄位註解應可直接成為規格示範，用來驗證現行程式是否實作完備。
	- 若註解與程式不一致，是開發疏忽，但不影響註解必要性。
- 其他補充原則
	- Inline 註解適用於不易理解的特殊邏輯，內容以解釋「為什麼」而非「做什麼」為主。
	- 變數、函式命名清楚可以減少註解，但不代表可以完全不寫。
	- 註解也是程式的一部分，需要維護，否則會與程式脫節造成誤導。
	- 若沒有同步機制，註解容易過期，因此更要寫在接近程式邏輯的位置並持續更新。

## 註解撰寫原則

- 註解需協助理解程式目的、流程與設計意圖，不是重複程式碼內容。
- 優先說明為什麼存在此邏輯，而不是描述程式在做什麼。
- 註解需與現行程式版本同步更新，避免過時造成維護成本。
- 團隊需建立統一英文程度的標準，專有名詞維持英文，其餘視需求加入母語補充。
- API、資料欄位、流程相關註解需能協助 QA、新人與外包快速對應客戶需求與功能稱呼。

## 語言規範

- 程式主註解以英文撰寫，專案名詞、Domain Term 保留英文原文。
- 需要協助外包或新人理解時，可加註中文補充在註解後方。
- 若功能名稱為客戶提供，需要原文呈現，以便串接與 QA 驗收對應。

## 何時需要寫註解

- 特殊邏輯、商業邏輯、非直觀流程。
- 行為受外部依賴（第三方 API、銀行規格、政府規範）。
- 命名無法完整表達流程背景或與需求規格對應時。
- 同一 function 內出現多段複雜邏輯，需要標記邊界或目的。
- API 欄位需標註用途、限制、客戶對應名稱。

## 何時可以不寫

- 程式碼命名已清楚可理解其行為。
- 標準語法、不涉及商業邏輯的基礎 CRUD。
- 註解僅重複敘述程式在做什麼。

## Inline 註解使用原則

- 用於標示關鍵判斷、例外行為、邊界條件。
- 內容著重說明為什麼，而非程式在做什麼。
- 放置於行前或行後皆可，但需維持一致風格。
